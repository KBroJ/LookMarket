## 📝 Q&A 작성 가이드

### 언제 작성하나요?

**작성 권장:**
- ✅ 3시간 이상 고민한 질문
- ✅ 같은 질문이 다시 생길 것 같을 때
- ✅ 면접에서 나올 법한 질문
- ✅ 팀원에게 설명할 만한 가치가 있는 질문

**작성 불필요:**
- ❌ 구글 검색 1분이면 나오는 질문
- ❌ 너무 프로젝트 특화된 일회성 질문
- ❌ 공식 문서에 잘 나와있는 질문

**번호 규칙:**
- 3자리 숫자 (001, 002, ...)

- 결번 없이

## 📊 작성 예시

### 좋은 Q&A 예시

**제목:**
```markdown
# Q: 왜 Domain에 @Entity를 사용하지 않나요?
```

**핵심 답변:**
```markdown
Domain을 프레임워크로부터 독립시켜 비즈니스 로직의 순수성을
보장하기 위해서입니다.
```

**상세 설명:**
- 구체적인 이유 3가지
- 코드 예시 (❌ 나쁜 예, ✅ 좋은 예)
- 실무 적용 사례

**길이:** 200-300줄 (적당함)

---

### 나쁜 Q&A 예시

**제목:**
```markdown
# Q: JPA 관련 질문
```
→ ❌ 너무 모호함

**핵심 답변:**
```markdown
그냥 그렇게 하는 게 좋습니다.
```
→ ❌ 이유 없음, 근거 없음

**설명 없음**
→ ❌ 상세 설명이 전혀 없음

**길이:** 10줄
→ ❌ 너무 짧아서 도움 안 됨

- 이유 없이 "그냥 그렇게 하세요"
- 예시 없이 추상적 설명만
- 검증되지 않은 추측
