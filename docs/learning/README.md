## 📄 현재 문서 목록

### Hexagonal-Architecture-Domain-구현-방식-비교.md

**목적**: 두 가지 Hexagonal Architecture 구현 방식 비교 (Domain에서 Lombok 사용 여부)

**주요 내용:**
1. **Domain에서 Lombok을 사용하지 않는 이유**
   - Setter의 근본적 문제
   - Behavior-rich Entity vs Anemic Domain Model
   - Getter는 왜 사용 가능한지

2. **Loopers vs LookMarket 아키텍처 비교**
   - 전체 구조 비교
   - Layer별 상세 비교 (Domain, Infrastructure, Application, API)
   - 변환 흐름 비교
   - 장단점 종합 분석

3. **설계 결정 요약**
   - LookMarket이 선택한 원칙
   - 면접 대비 답변 가이드
   - 최종 권장 사항

**언제 사용하나?**
- Hexagonal Architecture 이해가 필요할 때
- "왜 이렇게 설계했는가?" 질문 준비
- 다른 방식과의 차이점 학습

**특징:**
- Loopers 프로젝트: 실용적 접근 (Domain에 @Entity 사용)
- LookMarket 프로젝트: 순수 접근 (Domain과 Infrastructure 완전 분리)

### 멀티모듈-vs-폴더구분-비교분석.md

**목적**: 프로젝트 구조 방식(멀티 모듈 vs 폴더 구분)의 차이점과 선택 이유

**주요 내용:**
1. **멀티 모듈이란?**
   - 일반적인 오해 (패키지 구분과의 차이)
   - 독립적인 Gradle 프로젝트의 의미

2. **폴더 구분 vs 멀티 모듈 비교**
   - 각 방식의 특징 및 구조
   - 폴더 구분 방식의 문제점 (의존성 제어 불가능)
   - 멀티 모듈의 핵심 차이 (컴파일 타임 의존성 강제)

3. **왜 멀티 모듈을 선택했나?**
   - Hexagonal Architecture 강제
   - 빌드 시간 최적화
   - 재사용성
   - 테스트 격리
   - 포트폴리오 가치

4. **장단점 비교 및 다른 프로젝트 예시**
   - Spring Framework, Netflix 스타일 예시
   - 비교 표 (의존성 제어, 빌드, 테스트 등)

**언제 사용하나?**
- "왜 멀티 모듈을 사용하는가?" 질문에 답변 필요 시
- 폴더 구분 방식과 비교하고 싶을 때
- 면접 준비 (아키텍처 설계 결정 이유)

**특징:**
- 개념과 비교에 집중 (Why, What)
- 설정 방법은 `docs/guides/멀티모듈-설정-가이드.md` 참고

## 🎯 learning/ vs architecture/ 차이점

| 구분 | learning/ | architecture/ |
|------|----------|--------------|
| **목적** | 개인 학습 및 이해 | 프로젝트 원칙 정의 |
| **독자** | 본인, 학습자 | 팀원, 기여자 |
| **내용** | 비교 분석, Q&A | 강제 규칙, ADR |
| **톤** | "이런 방식들이 있다" | "우리는 이렇게 해야 한다" |
| **변경** | 자유롭게 | 신중하게 (팀 합의) |
| **예시** | "Loopers vs LookMarket 비교" | "Domain에 JPA 사용 금지" |

**간단히:**
- `learning/` = 공부하고 이해하기 위한 자료 (비교, 분석)
- `architecture/` = 프로젝트에서 지켜야 할 규칙 (원칙)

## 🎯 learning/ vs guides/ 차이점

| 구분 | learning/ | guides/ |
|------|----------|---------|
| **목적** | 개념 이해, 비교 | 사용법 안내 |
| **스타일** | 설명 및 분석 | 단계별 튜토리얼 |
| **내용** | "왜?", "차이는?" | "어떻게?" |
| **예시** | "멀티모듈 vs 폴더 구분 차이" | "멀티모듈 설정하는 방법" |

**간단히:**
- `learning/` = 이론, 비교, 분석 (why, what)
- `guides/` = 실습, 사용법 (how)

## 📖 사용 가이드

### 1. 새로운 개념 학습 시

**프로세스:**
1. 개념에 대해 공부
2. 여러 접근법 비교
3. `learning/`에 비교 분석 문서 작성
4. 프로젝트에 적용할 방식 결정
5. 결정 사항은 `architecture/decisions/`에 ADR 작성

**예시 흐름:**
```
1. Hexagonal Architecture 학습
   ↓
2. Loopers vs LookMarket 비교 분석
   (learning/Hexagonal-Architecture-Domain-구현-방식-비교.md)
   ↓
3. LookMarket 방식 선택
   ↓
4. 선택 이유 기록
   (architecture/decisions/ADR-001-순수-Domain-선택.md)
```

### 2. 질문이 생겼을 때

**Q&A 작성 프로세스:**
1. 질문을 명확히 정리
2. 답변 찾기 (검색, 실험, 문서)
3. `qna/` 폴더에 Q&A 문서 작성
4. 향후 같은 질문 시 빠른 참고

**작성 권장:**
- ✅ 3시간 이상 고민한 질문
- ✅ 나중에 또 궁금할 것 같은 질문
- ✅ 면접에서 나올 법한 질문

**작성 불필요:**
- ❌ 구글 검색 1분이면 나오는 질문
- ❌ 너무 프로젝트 특화된 질문

### 3. 다른 프로젝트와 비교 시

**비교 문서 작성:**
- 두 접근법의 장단점 객관적 비교
- 코드 예시 포함
- 실무 적용 시 고려사항
- 포트폴리오 관점 평가

## 🔍 자주 묻는 질문

### Q1: 공부한 내용을 모두 여기에 작성하나요?

**답변:** 아니요, 선택적으로 작성합니다.

**작성 권장:**
- ✅ 프로젝트와 직접 관련된 학습
- ✅ 나중에 다시 볼 만한 가치가 있는 내용
- ✅ 면접 준비에 도움되는 내용

**작성 불필요:**
- ❌ 기본적인 Java 문법
- ❌ 단순 라이브러리 사용법 (공식 문서 참고하면 됨)
- ❌ 일회성 실험

**기준:** "6개월 후에 다시 볼 것인가?"

---

### Q2: learning/과 guides/의 경계가 애매해요.

**답변:** 다음 기준으로 판단하세요.

**learning/에 작성:**
- "왜 이 방식을 선택했나?"
- "A 방식과 B 방식의 차이는?"
- "이 개념이 뭐지?"

**guides/에 작성:**
- "Docker Compose를 어떻게 설정하나?"
- "멀티모듈을 어떻게 만드나?"
- "이 도구를 어떻게 사용하나?"

**예시:**
- `learning/`: "멀티모듈 vs 폴더 구분의 차이와 각 장단점"
- `guides/`: "Gradle 멀티모듈 설정하는 방법 단계별 가이드"

---

### Q3: Q&A는 어느 정도 분량으로 작성하나요?

**답변:** 간결하게, 핵심만!

**구조:**
1. 질문 (1-2문장)
2. 핵심 답변 (3-5문장)
3. 상세 설명 (선택)
4. 예시 코드 (필요 시)

**길이:**
- 짧은 Q&A: 100-200줄
- 긴 Q&A: 300-500줄
- 너무 길면 별도 문서로 분리

### 비교 분석 문서 작성 시

**좋은 비교:**

```markdown

```
